# 06. Delivery Backlog

## 1. 우선순위 기준

- P0: MVP 검증에 반드시 필요
- P1: 추천 품질 또는 사용성에 큰 영향을 주지만 MVP 출시 후 보완 가능
- P2: 확장 기능 또는 운영 효율 개선

## 2. MVP Epic

### E1. 인증과 온보딩

| ID | Priority | Story | Acceptance Criteria |
| --- | --- | --- | --- |
| E1-1 | P0 | 사용자는 계정을 만들고 로그인할 수 있다. | 이메일 또는 소셜 로그인 중 하나가 동작한다. |
| E1-2 | P0 | 사용자는 기본 스타일 취향을 입력할 수 있다. | 선호/비선호 스타일 태그가 저장된다. |
| E1-3 | P0 | 사용자는 기본 위치와 알림 시간을 설정할 수 있다. | 위치 수동 입력과 알림 시간 저장이 가능하다. |
| E1-4 | P1 | 사용자는 권한을 나중에 허용할 수 있다. | 권한 거부 후에도 앱 핵심 기능이 막히지 않는다. |

### E2. 의류 등록

| ID | Priority | Story | Acceptance Criteria |
| --- | --- | --- | --- |
| E2-1 | P0 | 사용자는 의류 사진을 업로드할 수 있다. | 이미지가 storage에 저장되고 업로드 상태가 표시된다. |
| E2-2 | P0 | 앱은 의류 속성을 자동 추출한다. | 카테고리, 색상, 계절감, 스타일 태그가 반환된다. |
| E2-3 | P0 | 사용자는 AI 분석 결과를 수정할 수 있다. | 핵심 필드 수정 후 저장 가능하다. |
| E2-4 | P0 | 앱은 의류 일러스트를 생성한다. | 원본과 일러스트가 함께 표시된다. |
| E2-5 | P1 | 이미지 품질이 낮으면 재촬영을 안내한다. | 흐림/어두움/낮은 해상도 사유가 표시된다. |

### E3. 디지털 옷장

| ID | Priority | Story | Acceptance Criteria |
| --- | --- | --- | --- |
| E3-1 | P0 | 사용자는 등록한 의류 목록을 볼 수 있다. | 카테고리와 최근 등록순 목록이 제공된다. |
| E3-2 | P0 | 사용자는 의류 상세를 확인하고 수정할 수 있다. | 이미지와 속성 필드가 편집 가능하다. |
| E3-3 | P1 | 사용자는 의류 상태를 관리할 수 있다. | available, laundry, repair, storage 상태를 저장한다. |
| E3-4 | P1 | 사용자는 필터와 정렬을 사용할 수 있다. | 카테고리, 색상, 계절 필터가 작동한다. |

### E4. 추천

| ID | Priority | Story | Acceptance Criteria |
| --- | --- | --- | --- |
| E4-1 | P0 | 사용자는 원하는 느낌을 텍스트로 입력할 수 있다. | 요청 텍스트가 추천 요청으로 저장된다. |
| E4-2 | P0 | 앱은 자연어 요청을 구조화한다. | occasion, moodTags, constraints가 생성된다. |
| E4-3 | P0 | 앱은 보유 의류 기반 코디를 추천한다. | 최소 1개 코디 후보와 이유가 표시된다. |
| E4-4 | P0 | 앱은 날씨를 추천에 반영한다. | 기온/강수 조건에 맞지 않는 아이템이 제외된다. |
| E4-5 | P1 | 앱은 트렌드 시그널을 랭킹에 반영한다. | 추천 이유 또는 태그에 트렌드 반영 여부가 표시된다. |
| E4-6 | P1 | 사용자는 아이템 고정/제외 후 다시 추천받는다. | fixedItemIds/excludedItemIds가 적용된다. |

### E5. 아침 알림

| ID | Priority | Story | Acceptance Criteria |
| --- | --- | --- | --- |
| E5-1 | P0 | 사용자는 추천 알림 시간을 설정할 수 있다. | 설정 시간에 푸시 발송 대상이 된다. |
| E5-2 | P0 | 서버는 알림 전 오늘의 코디를 생성한다. | 추천 결과가 저장된 뒤 푸시가 발송된다. |
| E5-3 | P0 | 사용자는 푸시에서 추천 상세로 이동한다. | 딥링크가 추천 상세 화면을 연다. |
| E5-4 | P1 | 날씨 API 장애 시 대체 추천을 제공한다. | 캐시 또는 계절 규칙 기반 추천이 생성된다. |

### E6. 피드백과 착용 기록

| ID | Priority | Story | Acceptance Criteria |
| --- | --- | --- | --- |
| E6-1 | P0 | 사용자는 추천에 피드백을 남길 수 있다. | 좋아요/별로예요/덥다/춥다/튀다 사유가 저장된다. |
| E6-2 | P0 | 사용자는 추천 코디를 오늘 입음으로 기록할 수 있다. | WearLog가 생성되고 의류 착용 횟수가 증가한다. |
| E6-3 | P1 | 앱은 착용 기록을 캘린더로 보여준다. | 날짜별 착용 코디가 조회된다. |

## 3. 스프린트 계획 초안

### Sprint 0. Foundation

- 저장소 구조 생성
- 앱/서버 프로젝트 초기화
- 인증 방식 확정
- DB migration 기반 구축
- object storage 연결 방식 결정

### Sprint 1. Closet Registration

- 이미지 업로드
- 의류 분석 작업 생성
- AI 속성 추출
- 일러스트 생성
- 사용자 검수 및 저장

### Sprint 2. Digital Closet

- 옷장 목록
- 의류 상세
- 필드 수정
- 상태 관리
- 기본 필터

### Sprint 3. Recommendation

- 자연어 요청 해석
- 추천 후보 생성
- 날씨 API 연동
- 추천 결과 화면
- 피드백 저장

### Sprint 4. Morning Notification

- 알림 설정
- 예약 추천 생성
- 푸시 발송
- 딥링크 진입
- 장애 대체 로직

### Sprint 5. Hardening

- 개인정보 삭제 플로우
- AI 실패/재시도 UX
- 성능 개선
- 추천 품질 평가
- 베타 배포 준비

## 4. MVP 완료 조건

- 사용자가 계정을 만들고 의류를 1개 이상 등록할 수 있다.
- 등록된 의류는 일러스트와 속성 데이터로 저장된다.
- 사용자가 자연어 요청으로 추천을 받을 수 있다.
- 추천은 현재 날씨를 반영한다.
- 사용자가 지정한 시간에 오늘의 추천 알림을 받을 수 있다.
- 추천 결과에 피드백을 남길 수 있다.
- 사용자는 자기 데이터 삭제를 요청할 수 있다.
